# 技术文献

## 1. 文档说明

本文档用于说明 `canteen` 订餐系统的整体技术实现。该系统面向食堂点餐场景，采用前后端分离架构，包含：

- `Spring Boot` 后端服务
- `Vue 3` Web 管理端
- `微信小程序` 用户端

系统当前的核心目标是完成用户选座、点餐、下单、结算，以及管理员对订单、餐桌、菜品、用户和余额的维护管理。

## 2. 项目概述

### 2.1 系统定位

本系统是一个面向食堂场景的订餐管理系统，支持管理员通过网页后台维护业务数据，普通用户通过微信小程序完成选座、点餐、查看订单和个人资料管理。

### 2.2 角色划分

系统分为两类角色：

- `ADMIN`：管理员，主要使用 Web 管理端
- `USER`：普通用户，主要使用微信小程序端

### 2.3 核心能力

- 用户登录、注册与资料维护
- 餐桌占用、退桌与当前餐桌查询
- 菜品浏览、购物车下单、订单状态跟踪
- 订单结算与余额扣减
- 管理员对订单、餐桌、菜品、管理员、用户的后台维护
- 文件上传与图片访问

## 3. 技术架构

### 3.1 总体架构

系统采用典型三层结构：

1. 表现层
   - Web 管理端：基于 `Vue 3 + Element Plus`
   - 微信小程序端：基于微信小程序原生框架
   - 后端接口层：基于 `Spring Boot REST API`
2. 业务层
   - `Service` 负责登录校验、餐桌占用、订单生成、结算扣款、资料更新等业务逻辑
3. 数据访问层
   - `Mapper + MyBatis XML` 负责数据持久化与 SQL 执行

### 3.2 多端协作方式

- Web 管理端主要服务管理员
- 微信小程序端主要服务普通用户
- 两端共同复用后端接口与数据库
- 系统同时保留了部分旧接口和升级后的新接口，以兼容不同端的调用方式

### 3.3 接口响应规范

后端统一使用 `Result` 对象封装响应结果，返回结构如下：

```json
{
  "code": "200",
  "msg": "请求成功",
  "data": {}
}
```

## 7. 后端实现说明

### 7.1 通用模块

#### 7.1.1 `Result`

统一封装接口返回值，包括：

- `code`：状态码
- `msg`：消息说明
- `data`：返回数据

#### 7.1.2 `RoleEnum`

定义系统角色：

- `ADMIN`
- `USER`

#### 7.1.3 `AuthContext`

用于在请求生命周期中保存当前登录用户的：

- `userId`
- `role`

#### 7.1.4 `TokenUtils`

使用 `Hutool JWT` 生成和解析令牌，主要功能包括：

- 登录成功后签发 token
- 从 token 中解析用户 ID 和角色
- 统一设置 7 天有效期

#### 7.1.5 `AuthInterceptor`

后端通过拦截器读取请求头中的 `Authorization` 或 `token`，完成：

- token 解析
- 当前登录用户识别
- 请求结束后清理上下文

#### 7.1.6 `WebMvcConfig`

当前仅对部分用户侧新接口开启拦截保护：

- `/user/me`
- `/tables/current`
- `/orders`
- `/orders/my`
- `/orders/*/settle`

这意味着当前 Web 管理端仍主要依赖旧接口和页面侧角色控制，并未形成完整的后台 token 鉴权闭环。

### 7.2 认证接口

`WebController` 提供以下入口：

- `GET /`
  - 服务健康检查
- `POST /login`
  - 传统登录入口
  - 若未传 `role`，默认按管理员登录处理
- `POST /auth/login`
  - 用户侧登录入口
- `POST /auth/register`
  - 用户侧注册入口
- `POST /register`
  - 当前已不建议使用，直接返回提示信息

### 7.3 业务控制器

#### 7.3.1 `AdminController`

负责管理员的增删改查与分页查询。

#### 7.3.2 `UserController`

负责：

- 用户增删改查
- `GET /user/me` 查询当前用户资料
- `PUT /user/me` 更新当前用户资料

#### 7.3.3 `TablesController`

负责：

- 餐桌基础 CRUD
- `PUT /tables/addOrder` 旧版占桌接口
- `PUT /tables/removeOrder` 旧版退桌接口
- `GET /tables/current` 查询当前用户餐桌
- `POST /tables/current` 当前用户占桌
- `DELETE /tables/current` 当前用户退桌

#### 7.3.4 `FoodsController`

负责菜品的增删改查与分页查询。

#### 7.3.5 `OrdersController`

负责：

- 订单基础 CRUD
- `POST /orders` 当前用户创建订单
- `GET /orders/my` 查询当前用户订单
- `POST /orders/{id}/settle` 当前用户结算订单
- `GET /orders/income` 统计收入

#### 7.3.6 `FileController`

负责：

- 文件上传 `POST /files/upload`
- 文件访问 `GET /files/download/{fileName}`

上传文件实际写入仓库根目录下的 `photo/` 目录。

## 8. 服务层业务逻辑

### 8.1 `AdminService`

- 管理员登录校验
- 管理员新增时默认角色为 `ADMIN`
- 支持分页查询

### 8.2 `UserService`

- 用户登录
- 用户注册
- 用户新增
- 当前用户资料查询
- 当前用户资料更新

当前实现特点：

- 密码为明文比对
- 注册时角色固定为 `USER`
- 未实现独立修改密码接口

### 8.3 `TablesService`

核心规则包括：

- 一个用户同一时间只能占用一个餐桌
- 一个餐桌同一时间只能被一个用户占用
- 将餐桌状态从“是/否”字符串形式保存为空闲标志
- 支持当前用户直接占桌和退桌

### 8.4 `FoodsService`

主要提供菜品的基础增删改查，没有复杂业务逻辑。

### 8.5 `OrdersService`

核心业务包括：

- 生成订单编号 `orderNo`
- 写入下单时间
- 自动绑定当前用户 ID
- 查询当前用户订单
- 订单结算时扣减用户余额
- 统计“已完成”订单总收入

订单状态目前围绕三个阶段流转：

- `待出餐`
- `待结算`
- `已完成`

## 12. 关键业务流程

### 12.1 用户注册与登录

1. 用户在小程序输入账号和密码
2. 调用 `/auth/register` 或 `/auth/login`
3. 后端校验账号与密码
4. 登录成功后签发 token
5. 小程序将 token 和用户信息保存到本地

### 12.2 选座流程

1. 用户进入选座页
2. 调用 `/tables/current` 查询当前占用餐桌
3. 调用 `/tables/selectAll` 查询餐桌列表
4. 用户选择空闲餐桌
5. 调用 `POST /tables/current` 完成占桌

### 12.3 点餐下单流程

1. 用户进入点餐页
2. 加载当前餐桌和全部菜品
3. 用户将菜品加入购物车
4. 提交时将菜品拼接为 `content`
5. 调用 `POST /orders` 创建订单
6. 后端生成订单编号与下单时间

### 12.4 出餐与结算流程

1. 管理员在后台查看订单
2. 将订单状态从“待出餐”改为“待结算”
3. 用户在小程序订单页点击结算
4. 调用 `POST /orders/{id}/settle`
5. 后端校验用户身份与订单状态
6. 后端扣减用户余额并将订单改为“已完成”

### 12.5 用户资料与充值流程

1. 用户通过 `/user/me` 查看当前资料
2. 用户通过 `/user/me` 更新头像、姓名、手机号、性别
3. 管理员通过后台用户充值页面修改用户 `account`

## 15. 当前实现特点与局限

### 15.1 当前实现特点

- 已形成“后端 + Web 管理端 + 小程序用户端”的完整闭环
- 已实现 JWT 形式的用户侧登录鉴权
- 已实现当前餐桌绑定、订单结算和后台收入统计
- 已实现头像上传与图片资源访问
- 管理端和小程序端共用一套核心业务数据

### 15.2 当前局限

- 管理端鉴权仍偏轻量，未形成完整后台 token 校验体系
- 用户与管理员密码仍为明文存储与校验
- 订单内容仍以字符串保存，未拆分订单明细表
- 充值逻辑由前端计算后提交，缺少独立资金流水
- 文件上传缺少更严格的类型、大小与安全校验
- 缺少测试代码、初始化 SQL 和完整部署文档
