# 菜品分类需求整理

## 1. 文档目的

本文档用于在正式改代码前，先把“菜品分类”功能的需求范围、页面改动点、数据库变更点和验收标准理清楚，避免后续实现时反复返工。

本次需求暂不直接修改代码，只输出一份明确的实现说明。

## 2. 需求背景

当前系统的菜品仅包含以下信息：

- 名称 `name`
- 描述 `descr`
- 价格 `price`
- 图片 `img`

无论是管理员端的菜品管理，还是微信小程序端的点餐页面，目前都还没有“菜品分类”这个维度，因此用户无法按类别筛选菜品，管理员新增或编辑菜品时也无法指定类别。

## 3. 本次需求目标

为菜品增加“分类”字段，并在管理员端与用户端同时支持按分类展示和选择。

本次分类范围固定为 3 类：

- 素菜
- 荤菜
- 汤类

## 4. 需求范围

本次需求涉及 4 个部分：

1. 数据库 `foods` 表增加分类字段。
2. Spring Boot 后端的菜品实体、Mapper、接口查询能力增加分类支持。
3. Vue 管理员端的菜品管理页面增加分类展示、分类筛选、分类录入。
4. 微信小程序端的点餐页面增加分类下拉筛选。

本次需求不包含以下内容：

- 不新增“分类管理”表
- 不支持管理员自定义分类
- 不做多级分类
- 不做分类图标、分类图片等扩展设计
- 不调整订单结构

## 5. 分类字段设计建议

### 5.1 字段名称

建议在 `foods` 表新增字段：

- `type`

之所以建议使用 `type`，是因为含义直接，前后端字段名也统一，开发成本低。

如果你更偏向语义更明确的命名，也可以使用：

- `category`

但从当前项目风格看，`type` 更容易快速接入。

### 5.2 字段类型

建议数据库字段类型：

```sql
varchar(20)
```

可保存的值为：

- `素菜`
- `荤菜`
- `汤类`

### 5.3 是否允许为空

建议：

- 现有老数据迁移阶段可先允许为空
- 数据补齐后再改为非空

如果想一步到位，也可以新增时直接给默认值：

- 默认值：`素菜`

但从数据准确性角度，更推荐你手动补齐一次历史菜品分类。

## 6. 数据库需要修改的地方

### 6.1 需要修改的表

需要修改的表：

- `foods`

### 6.2 建议新增字段 SQL

```sql
alter table foods
add column type varchar(20) null comment '菜品分类';
```

### 6.3 历史数据补齐建议

如果表里已经有旧菜品数据，新增字段后，这些记录的 `type` 会是 `null`。建议手动补齐，例如：

```sql
update foods set type = '素菜' where id in (...);
update foods set type = '荤菜' where id in (...);
update foods set type = '汤类' where id in (...);
```

如果你想临时统一赋默认值，也可以先执行：

```sql
update foods set type = '素菜' where type is null;
```

### 6.4 如果后续希望设为非空

等历史数据补齐后，可再执行：

```sql
alter table foods
modify column type varchar(20) not null comment '菜品分类';
```

## 7. 后端需要改动的内容

### 7.1 实体类

文件：

- `springboot/src/main/java/com/example/entity/Foods.java`

需要新增字段：

- `private String type;`

并补充：

- `getType()`
- `setType()`

### 7.2 Mapper XML

文件：

- `springboot/src/main/resources/mapper/FoodsMapper.xml`

当前 `insert` 和 `update` 只处理：

- `name`
- `descr`
- `price`
- `img`

后续需要加入：

- `type`

同时，`selectAll` 和分页查询也要支持按分类筛选。

建议查询条件从目前的“仅按名称筛选”，升级为：

- 名称筛选 `name`
- 分类筛选 `type`

### 7.3 Mapper 接口

文件：

- `springboot/src/main/java/com/example/mapper/FoodsMapper.java`

当前查询方法大概率只传 `name`，后续要改成同时接收：

- `name`
- `type`

### 7.4 Service 层

文件：

- `springboot/src/main/java/com/example/service/FoodsService.java`

当前 `selectAll(name)`、`selectPage(name, ...)` 这类方法，需要扩展为支持分类参数，例如：

- `selectAll(name, type)`
- `selectPage(name, type, pageNum, pageSize)`

### 7.5 Controller 层

文件：

- `springboot/src/main/java/com/example/controller/FoodsController.java`

当前菜品接口查询参数只有名称，后续需要支持：

- `name`
- `type`

也就是说：

- 管理员端查询接口需要按名称和分类共同筛选
- 小程序端菜品列表接口也可以复用同一个查询接口

## 8. 管理员端需求整理

相关页面：

- `vue/src/views/manager/Foods.vue`

### 8.1 全局菜品信息区域

你提出的要求是：

- 在“全局菜品信息”这里出现一个下拉分类

可理解为：

- 在页面顶部展示所有菜品卡片的区域，增加一个分类下拉框
- 管理员选择分类后，全局菜品卡片按分类过滤显示

建议支持选项：

- 全部
- 素菜
- 荤菜
- 汤类

### 8.2 菜品管理查询区域

当前已有：

- 名称输入框查询

建议同步增加：

- 分类下拉查询

这样管理员在表格区域也可以组合筛选：

- 按名称搜索
- 按分类搜索

### 8.3 菜品列表表格

当前表格字段大致有：

- 名称
- 简介
- 价格
- 图片

后续应增加一列：

- 分类

这样管理员在表格里能直接看到每个菜属于哪一类。

### 8.4 新增菜品弹窗

你特别提出：

- 管理员的新增菜品也要多一项菜品分类的可点击选项

这部分应理解为：

- 新增菜品时，表单中增加“菜品分类”字段
- 交互形式使用下拉选择框 `el-select`

建议选项固定为：

- 素菜
- 荤菜
- 汤类

### 8.5 编辑菜品弹窗

既然新增有分类，那么编辑也必须能修改分类，否则后续无法维护。

因此编辑弹窗也应包含：

- 菜品分类下拉框

### 8.6 管理员端建议交互

建议默认行为如下：

- 查询默认选中“全部”
- 新增菜品时分类默认空值，要求管理员主动选择
- 如果你想减少漏选，也可以默认给“素菜”

更推荐：

- 必填
- 不允许空分类保存

## 9. 微信小程序端需求整理

相关页面：

- `wechat-miniprogram/pages/menu/index.js`
- `wechat-miniprogram/pages/menu/index.wxml`

### 9.1 点餐页增加分类下拉框

你提出的要求是：

- 用户端也在点餐的地方做一个下拉分类

建议放置位置：

- “今日菜品”标题附近
- 作为菜品列表上方的筛选条件

建议选项：

- 全部
- 素菜
- 荤菜
- 汤类

### 9.2 小程序端筛选逻辑

有两种实现方式：

1. 前端拿到全部菜品后在本地过滤
2. 每次切换分类时重新请求后端接口

结合当前项目规模，建议优先采用：

- 方案 1：先请求全部菜品，再在前端本地过滤

原因：

- 改动小
- 接口复用简单
- 当前分类只有 3 个，数据量通常不大

如果后续菜品数量很多，再切换成服务端筛选也很容易。

### 9.3 小程序页面展示建议

当前列表卡片里建议补充显示分类信息，可选做法：

- 在菜名下方显示“素菜 / 荤菜 / 汤类”
- 或使用一个小标签显示分类

这不是你本次明确要求的必须项，但建议一起做，用户识别会更清楚。

## 10. 接口层需求建议

虽然你这次没有要求新增新接口，但现有接口最好扩展查询参数。

### 10.1 现有接口可扩展为

- `GET /foods/selectAll?name=xxx&type=素菜`
- `GET /foods/selectPage?pageNum=1&pageSize=5&name=xxx&type=汤类`

### 10.2 新增/编辑接口提交字段

新增和编辑菜品时，请求体中需要增加：

```json
{
  "name": "番茄蛋汤",
  "descr": "口味清淡",
  "price": 8,
  "img": "/files/download/xxx.jpg",
  "type": "汤类"
}
```

## 11. 建议的实现顺序

建议按下面顺序开发，最稳妥：

1. 先改数据库 `foods.type`
2. 再改后端实体、Mapper、Service、Controller
3. 再改管理员端菜品管理页面
4. 最后改微信小程序点餐页面
5. 补齐旧数据分类并联调测试

这样可以避免前端先改好、后端字段却还没接通的情况。

## 12. 验收标准

功能完成后，至少应满足以下结果：

### 12.1 管理员端

- 菜品表格能看到“分类”列
- 全局菜品区域支持按分类筛选
- 查询区域支持按分类筛选
- 新增菜品时可以选择分类
- 编辑菜品时可以修改分类
- 保存后重新查询，分类信息不丢失

### 12.2 用户端

- 点餐页面出现分类下拉框
- 用户可按“全部 / 素菜 / 荤菜 / 汤类”筛选菜品
- 筛选后展示结果正确
- 加入购物车和下单流程不受影响

### 12.3 数据库与后端

- `foods` 表存在 `type` 字段
- 新增菜品时数据库能写入分类
- 编辑菜品时数据库能更新分类
- 查询菜品时接口能返回分类字段
